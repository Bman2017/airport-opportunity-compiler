# SpinOut U Core System Model v1

Status: **Frozen governing architecture**

Effective date: 2026-07-24

## Constitutional definition

SpinOut U is a governed opportunity configuration and deployment operating system. It acquires evidence from technologies, patents, research, institutions, markets, places, programs, and public sources; resolves that evidence into canonical objects and relationships; assembles compatible ingredients into configurations; evaluates which configurations constitute credible opportunities; designs product, licensing, venture, pilot, and deployment pathways; projects those opportunities into a discoverable marketplace; and learns from execution outcomes to improve every future configuration.

## Primary unit of invention

A **Configuration** is the primary unit of invention.

A Configuration is a governed assembly of problems, requirements, technologies, IP, software, capabilities, market actors, deployment environments, programs, constraints, and evidence that together define something that could be built.

## Governing lifecycle

```text
Source
  -> Evidence
  -> Canonical Object
  -> Canonical Relationship
  -> Configuration
  -> Opportunity
  -> Venture / Product / Service / Pilot / License / Program
  -> Build
  -> Deployment
  -> Outcome
  -> Learning
```

## Constitutional rules

1. Evidence and canonical truth are separate.
2. Every canonical entity has stable identity and provenance.
3. Every relationship is typed, directional, evidenced, confidence-scored, reviewable, and visibility-governed.
4. Semantic similarity is evidence for exploration, not proof of compatibility or opportunity.
5. A Configuration is an assembly; an Opportunity is an evaluated Configuration.
6. A Build is an activated Opportunity or Configuration with governed work, ownership, decisions, evidence, and next actions.
7. Deployment creates new Evidence, Outcomes, and Learning.
8. Public projections may never exceed the evidence, permissions, rights state, or review state beneath them.
9. Every new record may strengthen, weaken, complete, duplicate, contradict, supersede, or invalidate existing Configurations.
10. The frontend owns presentation, not canonical truth, opportunity generation, or relationship logic.

## Eight engines

### 1. Evidence Acquisition Engine

Acquires and preserves immutable source records, content hashes, source identity, permissions, versions, freshness, and provenance.

### 2. Canonical Understanding Engine

Normalizes records, resolves identities, detects duplicates and contradictions, classifies objects, and produces structured semantic and functional enrichment.

### 3. Knowledge Graph Engine

Persists and traverses evidence-backed relationships among actors, assets, knowledge, needs, environments, programs, configurations, opportunities, Builds, and outcomes.

### 4. Configuration Engine

Matches requirements to functions, validates compatibility, constructs bundles, identifies missing interfaces, creates alternatives, performs cross-domain transfer, and produces minimum viable and complete Configurations.

### 5. Opportunity Engine

Scores Configurations for evidence, technical coherence, market pull, rights clarity, novelty, deployment readiness, commercial potential, strategic fit, and completeness. It governs rejection, latency, review, publication, and activation readiness.

### 6. Venture Engine

Projects approved Configurations into products, services, software, licenses, pilots, programs, ventures, joint ventures, infrastructure projects, procurement responses, and commercialization models.

### 7. Deployment Engine

Determines pilot hosts, campus testbeds, corporate sponsors, integrators, procurement pathways, funding, regulatory requirements, site requirements, operators, and activation sequences.

### 8. Learning Engine

Converts pilot results, failures, costs, performance, buyer signals, licensing outcomes, decisions, and user actions into updated evidence, relationships, scores, configuration rules, and future opportunity rankings.

## Projection principle

Every marketplace rail, card, Opportunity Experience, Campus Experience, company page, spatial view, search result, recommendation, venture design, and Build seed is a governed projection of canonical objects and relationships.

## Permanent separation

```text
Evidence says what a source claims.
Canon says what the system currently recognizes.
Graph says how recognized objects relate.
Configuration says what can be assembled.
Opportunity says what is worth pursuing.
Venture says how value may be created.
Deployment says how it reaches reality.
Learning says what reality changed.
Projection says what a user may see and do.
```

## Reference implementation sequence

1. Freeze executable contracts.
2. Populate the 500-record governed reference corpus from the existing 52,000-record corpus and authoritative public sources.
3. Resolve identities and enrich ingredient interfaces.
4. Build evidence-backed relationships.
5. Implement Configuration Engine v1 recipes.
6. Evaluate, deduplicate, cluster, and review Opportunity candidates.
7. Generate venture and deployment projections.
8. Connect projections to the existing marketplace frontend.
9. Activate Builds.
10. Feed deployment outcomes into Learning.

## Acceptance thresholds for the reference implementation

- 500 governed source records
- 30 or more distinct institutions
- 15 or more domains
- 1,000 or more evidence-backed relationships
- 200 or more cross-institution relationships
- 150 or more cross-domain relationships
- 500 or more generated Configurations
- 200 or more Opportunity candidates
- 50 or more review-quality Opportunities
- 20 or more publishable Opportunities
- 60 or more venture designs
- 20 or more pilot or deployment designs
- 5 or more activation-ready Builds
- 100% relationship provenance coverage
- 100% public claim-policy compliance

These thresholds are implementation targets and may not be represented publicly until measured.

## Freeze rule

Future work must preserve this lifecycle, object separation, Configuration primacy, evidence governance, projection principle, and eight-engine architecture unless explicitly reopened through a governed architectural decision.
